# Роли и рабочий процесс (ты ↔ Codex ↔ Claude)

Кто что делает в этом проекте и как заставить Codex работать правильно.
Связано с: `IMPLEMENTATION-PLAN.md` (детальный runbook), `AGENTS.md` (правила
для Codex), спека в `docs/superpowers/specs/2026-06-17-trendscout-design.md`.

## Ментальная модель

> **Codex = исполнительный джун, который буквально делает то, что написано в
> `AGENTS.md` и в твоём промпте.** Пишет код, тесты, ветки, PR, гоняет `make`.
> **Ты = тимлид:** раздаёшь по одной задаче, проверяешь, и делаешь сам всё, что
> требует твоих аккаунтов, паролей, sudo или решений.
> **Claude = архитектор:** планирование, ревью спорных PR, исследование.

---

## 1. Общий цикл

Проект = 25 issue по слоям M1→M6. Работаешь по одной за раз:

```
открыть Codex → /impl-issue N → Codex пишет тест+код, гоняет make check, делает PR
   → ты смотришь PR → мержишь → следующая issue
```

Запуск (из обычного терминала, не Git Bash):
```powershell
cd C:\Users\k1lla\Desktop\ZeBrains-prac
codex
```
«Разогрев» в начале сессии:
```
Прочитай AGENTS.md, CLAUDE.md, docs/superpowers/specs/2026-06-17-trendscout-design.md
и IMPLEMENTATION-PLAN.md. Перескажи архитектуру, пул моделей под 6 ГБ и порядок
M1→M6. Ничего не меняй.
```
Затем: `/impl-issue 1`, далее `2`, `3`… по порядку зависимостей.

## 2. Что делает Codex

Всё внутри репозитория, не требующее твоих личных доступов:
- Код и тесты по TDD (сперва падающий тест).
- Ветка `feat/...`, conventional-commits.
- `make format` / `make check`.
- PR с `Closes #N`, проверка `gh pr checks`.
- Реализация issues M1–M5, Helm-чарты в M6.

**Три рычага управления:**
1. Вызывай **`/impl-issue N`** (не свободный текст) — промпт уже содержит TDD+гейт+PR.
2. Он сам читает **`AGENTS.md`**; если халтурит — пиши «**следуй AGENTS.md**».
3. **Одна issue = одна сессия/ветка.** Не давай «сделай M4 целиком».

## 4. Что делать «в случае чего»

| Симптом | Действие |
|---|---|
| `make check` красный | «make check падает, покажи вывод и почини, без зелёного не объявляй успех» |
| Codex выдумал API | «Проверь актуальный API через Context7 MCP, не угадывай» |
| Код без теста | «Нарушил TDD — сперва падающий тест. Переделай» |
| Полез во всю M4 сразу | Esc/Ctrl-C → «по одной issue: сейчас только #7» |
| Уперся в инфру (sudo/аккаунт) | твоя зона — сделай руками по runbook, верни Codex код |
| Модель медленно | 6 ГБ GPU — норма для 12B; проверь `ollama ps` |
| Git-конфликт | `git fetch origin && git rebase origin/main` или попроси Codex |
| Codex «потерял» контекст | новая сессия не помнит прошлую → снова «разогрев»-промпт |
| Забыл миграцию Drizzle | «закоммить снапшот миграции из drizzle» |
| Сомневаешься в правке | спроси Claude — разберёт diff/PR |

## 5. Как держать Codex на рельсах (отличие от Claude)

- Codex делает **ровно** то, что в промпте/`AGENTS.md` — своих «скиллов» у него нет.
- Всегда через **`/impl-issue N`**. Дрейфует → «следуй AGENTS.md».
- **Не разрешай авто-push без ревью**: пусть делает PR, merge жми сам.
- Память между сессиями **отсутствует** — источник истины это спека + issues +
  `IMPLEMENTATION-PLAN.md` (он их перечитывает).

## Шпаргалка «кто что»
- **Codex:** код, тесты, ветки, PR, `make`, чарты — по `/impl-issue N`.
- **Ты:** аккаунты (Supabase/Vercel), sudo/WSL/k3s, approve, merge, решения.
- **Claude:** планирование, ревью спорных PR, исследование, правка плана/issues.
