# ADR-004: API↔Client Contract via `@repo/shared`, Not Code Generation

**Date:** 2026-06-06
**Status:** Accepted
**Deciders:** @maxmrtnv

## Context

The API↔client contract must stay type-safe across the monorepo: when a request or response
shape changes on the backend, the frontend must see the change at compile time.

There are two common ways to achieve this:

1. **Code generation** — the backend emits an OpenAPI spec; a tool (`openapi-typescript`,
   `orval`, `swagger-typescript-api`, `openapi-generator`) generates a typed client the
   frontend imports.
2. **Shared source of truth** — both sides import the same hand-written schema module.

The repo is an all-TypeScript Turborepo where every consumer can import a workspace package
(`@repo/shared`). The only code generation present is Drizzle migrations (`make db-generate`).

How the contract works today, verified in the codebase:

- The contract is defined once as Zod schemas in `packages/shared/src/schemas/<domain>/`,
  with types derived via `z.infer` (e.g. `createOfferSchema` → `CreateOffer`).
- `apps/api` imports those schemas and validates at runtime with `.safeParse()`. Controllers
  accept `@Body() body: Record<string, unknown>`; there are **no `*.dto.ts` files** — the
  service validates against the shared schema (see
  `apps/api/src/modules/context-os/offers/offers.service.ts`).
- `packages/client-core` hooks consume the contract with `import type { XxxDto } from "@repo/shared"`
  and never redeclare it.

## Decision

**Keep the API↔client contract as a single source of truth: Zod schemas in `@repo/shared`,
imported by both the backend and the client. Do not add API client code generation.**

```text
packages/shared/src/schemas/<domain>/*.schema.ts
        │   export const createOfferSchema = z.object({ ... })
        │   export type   CreateOffer      = z.infer<typeof createOfferSchema>
        │
        ├──────────────► apps/api            import { createOfferSchema } from "@repo/shared"
        │   (backend)                        createOfferSchema.safeParse(body)   ← runtime validation
        │
        └──────────────► packages/client-core  import type { CreateOffer } from "@repo/shared"
            (frontend)                          ← compile-time types in hooks
```

A contract change is: update the shared Zod schema (`new-shared-schema` skill for new ones),
run `make test PKG=@repo/shared`, then `make typecheck` to surface breaking changes in every
consumer. This is what the `sync-api-contract` skill and the Flow B chains
(`new-api-module → sync-api-contract → new-client-hook`) actually do — sync, not generate.

For an all-TypeScript monorepo where every consumer can import a workspace package, the
shared-schema approach is strictly stronger than OpenAPI codegen:

| | Shared Zod (`@repo/shared`) | OpenAPI codegen |
|---|---|---|
| Schema translation steps | **0** (one module, both sides) | 2 (code → OpenAPI → generated client) |
| Runtime validation | Yes (`safeParse`) | No — types only; validation written separately |
| Drift risk frontend↔backend | None — same module imported | Continuous — generated artifact can lag the spec |
| Build step / committed artifact | None | Required |

Code generation exists to cross a **language or process boundary** (e.g. a Go backend with a
TS client, or external consumers who cannot import the package). The repo has no such boundary
today: all consumers are TS apps in the same Turborepo.

### When to revisit (codegen triggers)

Reconsider this decision — and adopt OpenAPI codegen generated *from* the Zod schemas for the
specific boundary — when a consumer appears that **cannot import `@repo/shared`**:

- a native mobile client (Swift/Kotlin),
- a service written in another language,
- an external/public API consumed by third parties.

## Consequences

### Positive

- The contract cannot silently drift — both sides import one file.
- Runtime validation and compile-time types come from the same definition.
- No codegen build step, no generated files to review or keep in sync.

### Negative

- Schema changes in `@repo/shared` have monorepo-wide blast radius; `make typecheck` after
  every contract change is mandatory.

### Neutral

- The OpenAPI spec served at `/api/docs` is generated by `@nestjs/swagger` from hand-written
  `@ApiResponse` decorators — it is **documentation only** and can drift from the Zod schemas.
  Deriving Swagger from Zod (`nestjs-zod` / `zod-to-openapi`) is a separate, tracked
  improvement (GitHub issue #182), not a change to this decision.
