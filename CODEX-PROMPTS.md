# Промпты для работы с Codex

Готовые к вставке промпты для сессии Codex по этому проекту. Копируй нужный.
Команды вида `/impl-issue` — это кастомные промпты из `~/.codex/prompts/`.
Связано: `ROLES-AND-WORKFLOW.md`, `IMPLEMENTATION-PLAN.md`, `AGENTS.md`.

> Правило №1: реализацию задач запускай через **`/impl-issue N`**, а не свободным
> текстом. Свободные промпты ниже — для отладки, ревью и нештатных ситуаций.

---

## 0. С чего начать (первая команда любой сессии)

**Разогрев — проверить, что Codex подтянул контекст:**
```
Прочитай AGENTS.md, CLAUDE.md, docs/superpowers/specs/2026-06-17-trendscout-design.md,
IMPLEMENTATION-PLAN.md и ROLES-AND-WORKFLOW.md. Кратко перескажи: архитектуру, пул
моделей под 6 ГБ VRAM, порядок задач M1→M6 и правило гейта перед push. Ничего не меняй —
только подтверди, что понял.
```
Если пересказал верно → следующая команда: `/impl-issue 1`.

---

## 1. Реализация задачи

**Основной способ:**
```
/impl-issue 1
```
(дальше `2`, `3`… по порядку зависимостей из IMPLEMENTATION-PLAN.md §2)

**Если хочешь добавить акцент к задаче:**
```
/impl-issue 6

Дополнительно: тест изоляции должен реально создавать двух пользователей и
проверять, что A получает 404/403 на отчёт B — не просто компиляция.
```

**Свободный вариант (если слэш-команда недоступна):**
```
Реализуй GitHub issue #N. Сначала прочитай AGENTS.md и спеку, найди секцию issue.
Строгий TDD: падающий тест → минимальный код → рефактор. Ветка feat/<имя> от свежего
main. Только объём этой issue. В конце: make format → make check → PR с "Closes #N".
```

---

## 2. Проверка / гейт

```
/verify
```
**Свободно:**
```
Прогони make format и make check. Покажи реальный вывод. Если красное — почини и
перепрогони. Не объявляй успех без зелёного вывода.
```

**Только посмотреть, не чиня:**
```
Прогони make check и покажи полный вывод ошибок. Пока ничего не чини — сначала
объясни, что именно падает и почему.
```

---

## 3. Когда что-то падает

**make check красный:**
```
make check падает. Покажи точный вывод. Найди корень причины (systematic debugging),
почини минимально, перепрогони гейт. Без зелёного результат не принимай.
```

**Билд-скрипты pnpm (better-sqlite3 / protobufjs):**
```
Если ошибка про better-sqlite3 или protobufjs — выполни `pnpm approve-builds`,
выбери эти пакеты, затем пересобери и перепрогони make check.
```

**Тест «зелёный, но подозрительно»:**
```
Этот тест проходит слишком легко. Покажи, что он реально падал ДО твоего кода
(red-фаза). Если нет — перепиши по TDD.
```

**Баг в существующем коде:**
```
Есть баг: <описание + как воспроизвести>. Сначала напиши падающий тест,
воспроизводящий баг, потом чини. Покажи red → green.
```

---

## 4. Когда Codex дрейфует (вернуть на рельсы)

**Нарушил TDD:**
```
Ты написал код без падающего теста — это нарушение TDD из AGENTS.md. Откати, начни
с теста, который падает, потом код.
```

**Полез делать слишком много:**
```
Стоп. Делай ТОЛЬКО issue #N, ничего сверх её скоупа. Остальное — отдельные issue.
```

**Выдумывает API библиотеки:**
```
Не угадывай API. Проверь актуальную документацию через Context7 MCP (resolve-library-id
→ query-docs) для <библиотека>, потом пиши код.
```

**Игнорит конвенции:**
```
Следуй AGENTS.md. В частности: <что нарушил — импорты/DI/Makefile/conventional commits>.
```

---

## 5. Git / GitHub

**Конфликт с main:**
```
Подтяни свежий main и сделай rebase ветки на origin/main, реши конфликты, перепрогони
make check, обнови PR.
```

**Забыл миграцию Drizzle:**
```
Проверь, что снапшот миграции Drizzle закоммичен (это частый провал CI). Если нет —
сгенерируй (make db-generate) и закоммить.
```

**Создать PR:**
```
Открой PR в main через gh, тело должно содержать "Closes #N" и краткое summary +
список тестов. Затем покажи gh pr checks.
```

---

## 6. Ревью перед merge (твоя самопроверка через Codex)

```
Покажи diff текущей ветки против main. Пройдись по нему как ревьюер: соответствует ли
acceptance-критериям issue #N, есть ли тесты на главную логику, нет ли нарушений
AGENTS.md (импорты, DI, TDD). Перечисли замечания, если есть.
```

---

## 7. Дизайн новой штуки (до кода)

```
/brainstorm <тема>
```
**Свободно:**
```
Не пиши код. Помоги спроектировать <что>. Задай уточняющие вопросы по одному, предложи
2-3 подхода с trade-offs, после согласования запиши спеку в docs/superpowers/specs/.
```
